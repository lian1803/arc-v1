핵심 마인드셋
사용자 중심 디자인(UCD)이란?
"내가 쓰고 싶은 거" 대신 "사용자가 필요한 거"를 만드는 사고방식
핵심 질문:
"이 기능이
진짜로
사용자의 문제를 해결하는가?"
기능을 많이 넣는 게 좋은 제품이 아닙니다. 사용자의
불편함
을 정확히 해결하는 게 좋은 제품입니다.
1
공감(Empathize)
사용자를 관찰하고 인터뷰합니다.
"언제 불편하세요?"
2
정의(Define)
문제를 한 문장으로 정리합니다.
"~할 때 ~가 불편하다"
3
해결(Ideate → Test)
아이디어 → 프로토타입 → 테스트.
빠르게 만들고 빠르게 검증합니다.
❌ UCD 없이 만들면
• 기능은 많은데 아무도 안 씀
• "이거 왜 여기 있어?" 반응
• 출시 후 전면 재설계
✅ UCD로 만들면
• 핵심 기능만 딱 있어서 편함
• "이거 내가 원하던 거!" 반응
• 최소 기능으로 빠르게 검증
우리가 배우는 IA → UJ → PRD 과정이 바로 UCD를 실천하는 방법입니다
디자인 감각
좋은 디자인이란? — 프롬프트에 녹여내기
디자인 = 예쁜 것이 아닙니다. 사용자가
"생각 없이 쓸 수 있는 것"
이 좋은 디자인입니다.
좋은 디자인 4원칙
1
명확성
— 보자마자 뭘 해야 하는지 안다
2
일관성
— 같은 동작은 같은 모양
3
피드백
— 누르면 반응이 온다
4
단순함
— 없어도 되는 건 빼라

📱

🎵

🛒
쿠팡

✂️
구글 검색
— 검색창 하나. 그게 전부.
프롬프트 Before → After
❌ Before: 기능만 나열
투두 앱을 만들어줘.
할 일을 추가하고 삭제하고
완료 표시 할 수 있게 해줘.
✅ After: UCD 원칙 반영
투두 앱을 만들어줘.
- 화면 중앙에 입력창, 엔터로 바로 추가
- 완료 시 체크 애니메이션 + 취소선
- 삭제는 스와이프 (실수 방지: 되돌리기 3초)
- 빈 상태: "오늘 할 일을 추가해보세요" 안내
- 색상: 화이트 배경, 포인트 #4F46E5
- 모바일 우선, 엄지 닿는 영역에 버튼 배치
같은 "투두 앱"이라도 사용자 경험을 고려한 프롬프트가 10배 좋은 결과를 만듭니다

기초 개념
프론트엔드란?
카페를 생각해보세요 — 손님이 보는 모든 것이 프론트엔드입니다
☕
카페 비유:
프론트엔드 = 카페 인테리어, 메뉴판, 카운터
손님(사용자)이 직접 눈으로 보고 손으로 만지는 모든 것
1
HTML
뼈대 · 건물 구조
<button>주문하기</button>
내용과 구조를 정의합니다
2
CSS
화장 · 인테리어
color: #ff0000;
색상, 크기, 위치를 꾸밉니다
3
JavaScript
동작 · 직원
button.onclick = order;
클릭, 움직임, 기능을 만듭니다
여러분이 AI Studio에서 만드는 건 = 프론트엔드! HTML · CSS · JS를 AI가 대신 써줍니다
기초 개념
백엔드란?
카페 주방 — 손님은 안 보지만 없으면 카페가 안 돌아갑니다
🍳 서버 = 주방장
요청을 받으면 요리(처리)해서 결과를 내보냅니다. 24시간 쉬지 않고 대기 중입니다.
📋 API = 주문서
프론트엔드가 백엔드에 요청하는 양식입니다. "아메리카노 1잔"처럼 규격이 정해져 있습니다.
🧊 데이터베이스 = 냉장고/창고
재료(데이터)를 체계적으로 보관합니다. 회원 정보, 주문 내역, 상품 목록 등을 저장합니다.
🪪 인증(Auth) = 직원 출입증
누가 어느 공간에 접근할 수 있는지 관리합니다. 로그인 · 회원가입이 여기에 해당합니다.
Lovable · Firebase Studio가 이 백엔드를 코딩 없이 자동으로 만들어줍니다!

실생활 API 예시
☀️ 날씨앱: "서울 날씨 알려줘" → API → 기상청 서버 → "맑음 15℃"

시멘틱(Semantic) HTML이란?
"의미가 있는 태그"를 쓰면 AI도, 브라우저도, 사람도 코드를 더 잘 이해합니다.
❌ 비시멘틱 (의미 없음)
<div class="header">
  <div class="logo">내 사이트</div>
  <div class="nav">메뉴1 메뉴2</div>
</div>
<div class="main">
  <div class="article">본문 내용</div>
</div>
<div class="footer">© 2025</div>
전부 <div> — 어디가 머리고 어디가 몸인지 모름
✅ 시멘틱 (의미 있음)
<header>
  <h1>내 사이트</h1>
  <nav>메뉴1 메뉴2</nav>
</header>
<main>
  <article>본문 내용</article>
</main>
<footer>© 2025</footer>
태그만 봐도 구조가 바로 보임!

실습 4 · 5분 · AI Studio
뽀모도로 타이머 앱 만들기
인터랙티브한 유틸리티 앱을 구조화된 프롬프트로 만듭니다.
Role: 시니어 프론트엔드 개발자
Task: 뽀모도로 타이머 웹앱 제작

Context:
- 용도: 집중력 향상을 위한 타이머
- 타겟: 시험 공부하는 대학생
- 핵심 개념: 25분 집중 → 5분 휴식 → 4세트 후 15분 긴 휴식

Features:
1. 원형 타이머 (남은 시간을 원형 프로그레스로 표시)
2. 시작 / 일시정지 / 리셋 버튼
3. 모드 표시: "집중 🔥" / "휴식 ☕" / "긴 휴식 🌴"
4. 자동으로 집중 → 휴식 전환 (알림 소리)
5. 오늘 완료한 세트 수 표시
6. 집중 시간 커스텀 설정 (15/25/30/45분)

- 스타일 참고: 미니멀 시계 앱
- 배경: #0f172a (다크), 텍스트: #ffffff
- 포인트 컬러: 집중=#ef4444 (빨강), 휴식=#10b981 (초록)
- 원형 타이머 크기: 화면 중앙, 300px
- 폰트: Pretendard, 숫자는 모노스페이스

Constraints:
- 모바일에서도 사용 가능 (반응형)
- 한국어 UI
- 브라우저 알림 퍼미션 요청
5원칙이 모두 적용된 프롬프트입니다. Role · Context · Task · Design · Constraints 확인!
수정 요청 잘 하는 법
AI에게 수정을 요청할 때 구체적일수록 결과가 좋습니다.
나쁜 수정 요청
"더 예쁘게 해줘"
"색깔 좀 바꿔줘"
"전체적으로 수정해줘"
→ 모호해서 AI가 엉뚱하게 수정
좋은 수정 요청
"헤더의 배경색을 #1e293b로 바꾸고,
카드 간격을 24px로 늘려줘"

"하단 탭바에 '알림' 탭을 추가하고
아이콘은 종 모양으로 해줘"
→ 위치 + 속성 + 값이 명확
Tip 1:
한 번에 하나씩 수정 요청 — 여러 개를 한꺼번에 하면 AI가 혼란
Tip 2:
위치 + 속성 + 값을 명시 — "어디의 무엇을 어떻게"
Tip 3:
변경 전/후를 비교 확인 — 의도대로 바뀌었는지 검증

MBTI 스타일 심리 테스트 만들기

기능:
- 10개의 2지선다 질문 (A or B)
- 각 질문은 카드 형태로 한 장씩 표시

User Journey란?
사용자가 서비스를 발견 → 사용 → 재방문하는 전체 흐름. 이 흐름의 각 단계가 나중에

이 됩니다.
작성법
1. 발견/인지
— 사용자가 문제를 느끼는 순간
2. 진입
— 앱 설치, 가입
3. 핵심 경험
— 메인 기능을 처음 사용하는 순간
4. 일상 사용
— 반복적으로 사용하는 패턴
5. 확장/습관화
— 고급 기능 발견, 재방문
예시: 프레시키퍼 User Journey
1. 발견: "냉장고에 뭐 있었지... 또 버리게 생겼네"
   → 감정: 짜증, 죄책감

2. 진입: 앱 설치 → Google 로그인
   → 필요 기능: 회원가입/로그인

3. 첫 등록: 냉장고 식재료 사진 촬영 → 자동 인식
   → 필요 기능: 식재료 등록 (CRUD - Create)

4. 일상 관리: D-day 알림 확인 → 임박 재료 확인
   → 필요 기능: 알림, 목록 조회 (Read)

5. 활용: 임박 재료로 레시피 검색
   → 필요 기능: 레시피 추천

6. 소비: 사용 완료 체크 / 폐기 처리
   → 필요 기능: 수정/삭제 (Update/Delete)

7. 습관화: 주간 절약 리포트 확인
   → 필요 기능: 통계

Git = 코드 변경 기록을 자동으로 관리해주는 도구. "저장"의 상위 버전이라고 생각하세요!
핵심 명령어 4가지
Git 핵심 개념 — 쉽게 이해하기
택배 보내기에 비유하면 이해가 쉽습니다
1
commit (커밋) = "박스에 담기"
변경한 코드를 하나의 박스에 묶어서 설명 메모를 붙이는 것
# 예시: "로그인 기능 추가" 라고 메모를 남기며 저장
git commit -m "로그인 기능 추가"
2
push (푸시) = "택배 보내기"
내 컴퓨터의 코드 박스를 GitHub(인터넷 창고)에 보내는 것
# 내 코드를 GitHub에 업로드
git push
3
pull (풀) = "택배 받기"
GitHub에 있는 최신 코드를 내 컴퓨터로 가져오는 것
# 팀원이 올린 최신 코드를 내려받기
git pull
4
clone (클론) = "복사해오기"
GitHub에 있는 프로젝트를 처음으로 내 컴퓨터에 통째로 복사하는 것
# GitHub의 프로젝트를 내 컴퓨터로 복제
git clone https://github.com/유저이름/프로젝트이름

Repository (저장소)
프로젝트 하나 = 저장소 하나. 코드 + 변경기록이 모두 여기에 보관됩니다.

비유: 앱 = 식당
프론트엔드 = 식당 인테리어 (손님이 보는 것)
Supabase = 주방 + 냉장고 + 금고
주방(서버)에서 요리하고, 냉장고(DB)에 재료를 보관하고, 금고(Auth)가 출입을 관리합니다.
왜 Supabase?
무료 플랜으로 충분히 시작 가능
Lovable이 자동으로 연결해줌
SQL 몰라도 Table Editor로 관리
구글 로그인 등 간단히 설정
지금은 이해만 하면 됩니다! Lovable이 Supabase를 자동으로 연결해줘요.

핵심 팁 5가지
Lovable 잘 쓰는 비법
실전에서 얻은 노하우 — 이것만 기억하세요
1. Plan 모드 활용하기
채팅창 옆의
Plan 모드(파란색)
를 켜면 AI가 먼저 계획을 세운 후 코딩합니다. 크레딧 1개 추가 소모되지만, 더 좋은 결과물이 나옵니다.
"바로 코딩"이 아니라 "계획 → 코딩" 순서로 진행됨
2. 한 번에 하나씩 요청
여러 기능을 한꺼번에 요청하면 AI가 혼란스러워하거나 빠뜨립니다.
하나의 요청 = 하나의 기능 변경
이 원칙!
❌ "수정 버튼 넣고 디자인도 바꾸고 DB도 연결해줘"
✅ "수정 버튼을 넣어줘"  →  (완료 후)  →  "디자인 바꿔줘"
3. 에러 = 스크린샷 디버깅
에러가 나면 당황하지 말고 그 화면을
스크린샷
으로 찍어서 Lovable 채팅에 붙여넣기. AI가 에러를 보고 직접 수정합니다.
"초대 코드를 입력했는데 이 에러가 떠 [스크린샷 첨부]. 수정해줘"
4. 모바일 뷰 꼭 확인
오른쪽 위
디바이스 전환 버튼
(랩탑→태블릿→모바일)으로 항상 모바일 뷰를 확인하세요. 대부분의 사용자는 모바일입니다!
"모바일 화면에서 레이아웃이 깨져 있어 [스크린샷 첨부]. 최적화해줘"
5. Visual Edit 모드 — 특정 영역만 수정
채팅 대신
Visual Edit 버튼
을 누르면 화면에서 직접 요소를 클릭해서 그 부분만 수정 요청을 할 수 있습니다. 작은 디자인 수정에 매우 편리!
Publish(배포) 전에 꼭 Security 탭에서 무료 보안 스캔을 실행하세요. 실제 사용자에게 배포하기 전 필수!

Preview vs Publish
Preview:
개발 중 미리보기. Lovable 안에서만 확인 가능. 일부 기능이 작동하지 않을 수 있음.
Publish:
실제 배포. 고유 URL이 생기고 누구나 접속 가능. DB 연동도 이때 정상 작동.
배포 후 수정하면?
코드 수정 후 다시 Publish를 누르면 최신 버전으로 자동 업데이트됩니다.
이전 버전은 History에서 확인 가능!
Preview에서 안 되던 기능이 Publish 후에는 작동하는 경우가 많아요. 에러가 나면 Publish 먼저 해보세요!

3
프로젝트 선택 또는 새로 생성

4
자동으로 API 키 연결 + 테이블 생성 완료!
DB 스키마까지 자동으로 만들어줍니다
수동 연결 (오류 시)

3
anon/public key 복사
eyJh... 으로 시작하는 긴 키

Lovable 채팅에 입력
"Supabase URL은 [URL]이고 anon key는 [KEY]야"
대부분 자동 연결됩니다! 수동은 오류가 날 때만 사용하세요.
가이드 · Supabase
Supabase Table Editor 활용법
Lovable이 만든 테이블을 직접 확인하고 수정하는 방법
Table Editor 기본 사용법
1
supabase.com 로그인 → 프로젝트 선택
2
좌측 메뉴 → Table Editor 클릭
3
생성된 테이블 목록 확인
listings, reviews, bookings 등
4
행 클릭 → 데이터 직접 수정 가능
5
"+ Insert Row" → 새 데이터 직접 추가
6
컬럼 헤더 클릭 → 정렬/필터
자주 쓰는 조작
테이블 구조 보기
Columns 탭에서 필드명, 타입 확인. 어떤 데이터가 저장되는지 한눈에 볼 수 있습니다.
RLS (Row Level Security)
Authentication → Policies에서 보안 설정. "내 데이터만 보기" 같은 규칙을 설정합니다.
SQL 직접 실행
SQL Editor에서 쿼리 직접 작성 가능. 복잡한 조회나 데이터 수정에 유용합니다.
데이터 내보내기
CSV로 다운로드 가능. 엑셀에서 데이터를 확인하거나 백업할 수 있습니다.
Lovable이 자동 생성한 테이블을 Table Editor에서 확인하면 DB 구조를 이해할 수 있어요!

구글이 만든 올인원 백엔드 플랫폼

Firestore
NoSQL 데이터베이스. 문서(document) 안에 데이터를 JSON처럼 저장합니다. 폴더 속 파일 구조로 이해하면 쉽습니다.
2
Authentication
구글 · 이메일 로그인을 한번에 구현합니다. Firebase Console에서 몇 번의 클릭으로 활성화됩니다.
3
Hosting
만든 앱을 인터넷에 공개합니다. 자동으로 URL이 부여되어 누구나 접속할 수 있습니다.
4
Cloud Functions
서버 코드를 클라우드에서 실행합니다. 결제 처리, 이메일 발송 등 고급 기능에 사용합니다.

'오늘할일' - 할 일 관리 웹앱을 만들어줘.

주요 기능:
1. 사용자 로그인 (Google Auth)
2. 할 일 추가 (제목, 마감일, 우선순위: 높음/중간/낮음)
3. 할 일 목록 (미완료/완료 탭 분리)
4. 완료 체크 토글 (취소선 애니메이션)
5. 할 일 수정/삭제
6. 우선순위별 색상 표시 (높음: 빨강, 중간: 주황, 낮음: 초록)
7. 오늘 완료한 개수 표시

기술 스택: Next.js + Firebase (Firestore + Auth)
디자인: 미니멀, 흑백 기반, 깔끔한 체크리스트 UI
