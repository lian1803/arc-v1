사용자 중심 디자인(UCD)이란?

"내가 쓰고 싶은 거" 대신 "사용자가 필요한 거"를 만드는 사고방식

핵심 질문: "이 기능이 진짜로 사용자의 문제를 해결하는가?"

기능을 많이 넣는 게 좋은 제품이 아닙니다. 사용자의 불편함을 정확히 해결하는 게 좋은 제품입니다.

1

공감(Empathize)

사용자를 관찰하고 인터뷰합니다.
"언제 불편하세요?"

2

정의(Define)

문제를 한 문장으로 정리합니다.
"~할 때 ~가 불편하다"

3

해결(Ideate → Test)

아이디어 → 프로토타입 → 테스트.
빠르게 만들고 빠르게 검증합니다.

❌ UCD 없이 만들면

• 기능은 많은데 아무도 안 씀

• "이거 왜 여기 있어?" 반응

• 출시 후 전면 재설계

✅ UCD로 만들면

• 핵심 기능만 딱 있어서 편함

• "이거 내가 원하던 거!" 반응

• 최소 기능으로 빠르게 검증

우리가 배우는 IA → UJ → PRD 과정이 바로 UCD를 실천하는 방법입니다
디자인 감각

좋은 디자인이란? — 프롬프트에 녹여내기

디자인 = 예쁜 것이 아닙니다. 사용자가 "생각 없이 쓸 수 있는 것"이 좋은 디자인입니다.

좋은 디자인 4원칙

1
명확성 — 보자마자 뭘 해야 하는지 안다
2
일관성 — 같은 동작은 같은 모양
3
피드백 — 누르면 반응이 온다
4
단순함 — 없어도 되는 건 빼라

실제 서비스 예시

📱 카카오톡 송금 — 금액 입력 → 비밀번호 → 끝. 3단계.

🎵 Spotify — 재생 버튼이 항상 같은 위치.

🛒 쿠팡 — "장바구니 담김!" 토스트 알림.

✂️ 구글 검색 — 검색창 하나. 그게 전부.

프롬프트 Before → After

❌ Before: 기능만 나열

투두 앱을 만들어줘. 할 일을 추가하고 삭제하고 완료 표시 할 수 있게 해줘.

✅ After: UCD 원칙 반영

투두 앱을 만들어줘. - 화면 중앙에 입력창, 엔터로 바로 추가 - 완료 시 체크 애니메이션 + 취소선 - 삭제는 스와이프 (실수 방지: 되돌리기 3초) - 빈 상태: "오늘 할 일을 추가해보세요" 안내 - 색상: 화이트 배경, 포인트 #4F46E5 - 모바일 우선, 엄지 닿는 영역에 버튼 배치
같은 "투두 앱"이라도 사용자 경험을 고려한 프롬프트가 10배 좋은 결과를 만듭니다

프로토타입 · 데모 · MVP · PoC

"다 비슷해 보이지만 목적이 완전히 다릅니다"

프로토타입 Prototype

목적: 아이디어를 눈에 보이는 형태로 만들기

• 동작하는 화면 시안 (클릭 가능)

• "이런 느낌의 앱이에요" 확인용

• 오늘 우리가 만드는 것 = 프로토타입

데모 Demo

목적: 고객/투자자에게 "이거 됩니다" 보여주기

• 프로토타입을 보여주는 행위

• "저희 서비스 한번 보세요!" 발표

• 프로토타입 ≈ 결과물, 데모 ≈ 발표 과정

MVP Minimum Viable Product

목적: 핵심 기능만 넣어 시장에서 검증

• 실제 사용자가 쓸 수 있는 수준

• 로그인·결제 등 백엔드 포함

• "사람들이 진짜 쓸까?" 테스트

PoC Proof of Concept

목적: "이 기술/방법이 가능한가?" 증명

• 기술적 실현 가능성 확인

• 디자인·UX는 신경 안 씀

• 예: "AI가 음식 사진 인식 가능?" 테스트

한눈에 흐름 정리

PoC → 기술 검증 | 프로토타입 → 형태 확인 | 데모 → 발표·설득 | MVP → 시장 검증
린 스타트업: "빠르게 만들고 → 빠르게 검증 → 빠르게 개선"하는 반복 과정. 이 수업이 첫 걸음입니다!

프론트엔드란?

카페를 생각해보세요 — 손님이 보는 모든 것이 프론트엔드입니다

☕ 카페 비유: 프론트엔드 = 카페 인테리어, 메뉴판, 카운터
손님(사용자)이 직접 눈으로 보고 손으로 만지는 모든 것

1

HTML

뼈대 · 건물 구조

<button>주문하기</button>

내용과 구조를 정의합니다

2

CSS

화장 · 인테리어

color: #ff0000;

색상, 크기, 위치를 꾸밉니다

3

JavaScript

동작 · 직원

button.onclick = order;

클릭, 움직임, 기능을 만듭니다

기초 개념

백엔드란?

카페 주방 — 손님은 안 보지만 없으면 카페가 안 돌아갑니다

🍳 서버 = 주방장

요청을 받으면 요리(처리)해서 결과를 내보냅니다. 24시간 쉬지 않고 대기 중입니다.

📋 API = 주문서

프론트엔드가 백엔드에 요청하는 양식입니다. "아메리카노 1잔"처럼 규격이 정해져 있습니다.

🧊 데이터베이스 = 냉장고/창고

재료(데이터)를 체계적으로 보관합니다. 회원 정보, 주문 내역, 상품 목록 등을 저장합니다.

🪪 인증(Auth) = 직원 출입증

누가 어느 공간에 접근할 수 있는지 관리합니다. 로그인 · 회원가입이 여기에 해당합니다.

Lovable · Firebase Studio가 이 백엔드를 코딩 없이 자동으로 만들어줍니다!

→ 앱(프론트)이 API(주문서)를 통해 서버(주방)에 요청 → 서버가 DB(냉장고)에서 재료 꺼내 처리 → 결과를 다시 앱에 전달

→ 저장

시멘틱(Semantic) HTML이란?

"의미가 있는 태그"를 쓰면 AI도, 브라우저도, 사람도 코드를 더 잘 이해합니다.

❌ 비시멘틱 (의미 없음)

<div class="header"> <div class="logo">내 사이트</div> <div class="nav">메뉴1 메뉴2</div> </div> <div class="main"> <div class="article">본문 내용</div> </div> <div class="footer">© 2025</div>

전부 <div> — 어디가 머리고 어디가 몸인지 모름

✅ 시멘틱 (의미 있음)

<header> <h1>내 사이트</h1> <nav>메뉴1 메뉴2</nav> </header> <main> <article>본문 내용</article> </main> <footer>© 2025</footer>

태그만 봐도 구조가 바로 보임!

시멘틱 태그역할비유
<header>페이지 상단 (로고, 메뉴)책의 표지
<nav>내비게이션 메뉴책의 목차
<main>핵심 본문 영역책의 본문
<article>독립적인 콘텐츠책의 한 챕터
<section>관련 콘텐츠 묶음챕터 안의 소단원
<footer>페이지 하단 (저작권, 링크)책의 뒤표지
프롬프트에 "시맨틱 HTML 구조 사용"이라고 쓰면 AI가 알아서 의미 있는 태그로 만들어줍니다. 외울 필요 없어요!

내 자기소개 페이지 만들기

5원칙을 적용한 고도화 프롬프트로 자기소개 페이지를 만듭니다.

CASE A: 아이디어가 있는 경우
Role: 시니어 프론트엔드 개발자 Task: '프레시키퍼' 앱의 메인 대시보드 웹페이지 제작 Context: - 서비스: 냉장고 식재료 유통기한을 AI로 관리하고, 남은 재료로 레시피 추천 - 타겟: 25-35세 1인 가구 직장인 - 핵심 기능: 식재료 목록, 레시피 추천 버튼, 알림 설정 Design: - 스타일: 깔끔한 카드 UI, 친환경 느낌 - 메인 컬러: #10b981 (초록), 배경: #ffffff - 폰트: Pretendard - 모바일 반응형 Sections: 1. 헤더: 로고 + 알림 아이콘 2. 식재료 카드 리스트 (이름, D-day, 상태 색상) 3. 하단 탭바: 홈, 스캔, 레시피, 설정 샘플 데이터 5개를 포함. 한국어 UI.
CASE B: 아이디어가 없는 경우
Role: 시니어 프론트엔드 개발자 Task: '하루습관' 습관 트래커 앱의 메인 화면 제작 Context: - 서비스: 매일 실천할 습관을 체크하고 주간 통계를 보여주는 앱 - 타겟: 자기계발에 관심 있는 20대 - 핵심 기능: 습관 체크리스트, 주간 달성률 차트, 동기부여 문구 Design: - 스타일: 미니멀, 부드러운 느낌 - 메인 컬러: #8b5cf6 (보라), 배경: #ffffff - 폰트: Pretendard - 모바일 반응형 Sections: 1. 헤더: 날짜 + 인사말 2. 오늘의 습관 카드 (토글 체크, 아이콘) 3. 주간 달성률 바 차트 4. 하단 탭바: 오늘, 통계, 설정 샘플 습관 5개 포함. 한국어 UI.

AI에게 수정을 요청할 때 구체적일수록 결과가 좋습니다.

나쁜 수정 요청

"더 예쁘게 해줘" "색깔 좀 바꿔줘" "전체적으로 수정해줘"

좋은 수정 요청

"헤더의 배경색을 #1e293b로 바꾸고, 카드 간격을 24px로 늘려줘" "하단 탭바에 '알림' 탭을 추가하고 아이콘은 종 모양으로 해줘"

→ 위치 + 속성 + 값이 명확

Tip 1: 한 번에 하나씩 수정 요청 — 여러 개를 한꺼번에 하면 AI가 혼란
Tip 2: 위치 + 속성 + 값을 명시 — "어디의 무엇을 어떻게"
Tip 3: 변경 전/후를 비교 확인 — 의도대로 바뀌었는지 검증

User Journey란?

사용자가 서비스를 발견 → 사용 → 재방문하는 전체 흐름. 이 흐름의 각 단계가 나중에 기능 목록(Features)이 됩니다.

작성법

1. 발견/인지 — 사용자가 문제를 느끼는 순간
2. 진입 — 앱 설치, 가입
3. 핵심 경험 — 메인 기능을 처음 사용하는 순간
4. 일상 사용 — 반복적으로 사용하는 패턴
5. 확장/습관화 — 고급 기능 발견, 재방문
예시: 프레시키퍼 User Journey
1. 발견: "냉장고에 뭐 있었지... 또 버리게 생겼네" → 감정: 짜증, 죄책감 2. 진입: 앱 설치 → Google 로그인 → 필요 기능: 회원가입/로그인 3. 첫 등록: 냉장고 식재료 사진 촬영 → 자동 인식 → 필요 기능: 식재료 등록 (CRUD - Create) 4. 일상 관리: D-day 알림 확인 → 임박 재료 확인 → 필요 기능: 알림, 목록 조회 (Read) 5. 활용: 임박 재료로 레시피 검색 → 필요 기능: 레시피 추천 6. 소비: 사용 완료 체크 / 폐기 처리 → 필요 기능: 수정/삭제 (Update/Delete) 7. 습관화: 주간 절약 리포트 확인 → 필요 기능: 통계

실전에서 얻은 노하우 — 이것만 기억하세요

1. Plan 모드 활용하기

채팅창 옆의 Plan 모드(파란색)를 켜면 AI가 먼저 계획을 세운 후 코딩합니다. 크레딧 1개 추가 소모되지만, 더 좋은 결과물이 나옵니다.

"바로 코딩"이 아니라 "계획 → 코딩" 순서로 진행됨

2. 한 번에 하나씩 요청

여러 기능을 한꺼번에 요청하면 AI가 혼란스러워하거나 빠뜨립니다. 하나의 요청 = 하나의 기능 변경이 원칙!

❌ "수정 버튼 넣고 디자인도 바꾸고 DB도 연결해줘" ✅ "수정 버튼을 넣어줘" → (완료 후) → "디자인 바꿔줘"

3. 에러 = 스크린샷 디버깅

에러가 나면 당황하지 말고 그 화면을 스크린샷으로 찍어서 Lovable 채팅에 붙여넣기. AI가 에러를 보고 직접 수정합니다.

"초대 코드를 입력했는데 이 에러가 떠 [스크린샷 첨부]. 수정해줘"

4. 모바일 뷰 꼭 확인

오른쪽 위 디바이스 전환 버튼(랩탑→태블릿→모바일)으로 항상 모바일 뷰를 확인하세요. 대부분의 사용자는 모바일입니다!

"모바일 화면에서 레이아웃이 깨져 있어 [스크린샷 첨부]. 최적화해줘"

5. Visual Edit 모드 — 특정 영역만 수정

채팅 대신 Visual Edit 버튼을 누르면 화면에서 직접 요소를 클릭해서 그 부분만 수정 요청을 할 수 있습니다. 작은 디자인 수정에 매우 편리!

Publish(배포) 전에 꼭 Security 탭에서 무료 보안 스캔을 실행하세요. 실제 사용자에게 배포하기 전 필수!

Preview vs Publish

Preview: 개발 중 미리보기. Lovable 안에서만 확인 가능. 일부 기능이 작동하지 않을 수 있음.

코드 수정 후 다시 Publish를 누르면 최신 버전으로 자동 업데이트됩니다.
